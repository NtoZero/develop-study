# Make Study Content

검증된 기술 조사 자료를 독자가 혼자 읽어도 원리를 유도하고 계산하며 실패를 진단할 수 있는 교재형 과정으로 변환하라. 단순히 요약하거나 조사 목차를 바꾸지 말고 개념 간 의존 관계와 난이도 상승을 분석해 장별 논증 구조를 새로 설계하라.

## 역할 경계

- `docs/research/<domain>/<topic>`의 문서를 사실과 근거의 기준으로 사용하라.
- 입력과 출력의 `<domain>/<topic>` 경로를 정확히 일치시켜 `docs/research`와 `docs/study`가 같은 주제를 일대일로 찾을 수 있게 하라.
- 이 스킬은 조사 결과를 교육적으로 가공하는 역할에 집중하라.
- 필요한 핵심 근거가 없거나 버전이 맞지 않으면 기억으로 채우지 마라. 프로젝트의 `$tech-research` 스킬을 사용할 수 있으면 먼저 조사 자료를 보강하고, 사용할 수 없으면 부족한 근거와 영향을 사용자에게 알려라.
- 조사 문서의 추론, 정신 모델, 비유 표시는 학습 자료에서도 유지하라. 비유를 실제 구현처럼 설명하지 마라.
- 조사 자료의 직접 출처 링크와 버전 조건을 학습 자료까지 추적 가능하게 보존하라.
- 기존 `practice/<domain>/<topic>`이 있으면 실행 방법과 관찰 가능한 결과를 읽고 관련 학습 문서에 연결하되, 이 스킬에서 실습 프로젝트 자체를 구현하지 마라. 실습 구현은 `$prepare-practice`의 역할이다.

## 기본 입출력

- 사용자가 별도로 지정하지 않으면 한국어로 작성하라.
- 입력 경로를 `docs/research/<domain>/<topic>/`으로 간주하고 `index.md`와 번호가 붙은 모든 조사 문서를 읽어라.
- 출력 경로를 입력과 동일한 `<domain>/<topic>`을 사용한 `docs/study/<domain>/<topic>/`으로 구성하라.
- 전체 과정과 탐색 목차를 `index.md`에 작성하되 실제 학습 본문을 종합해 넣지 마라.
- 상세 학습 자료를 `01-<lesson-slug>.md`, `02-<lesson-slug>.md`처럼 학습 순서가 드러나는 이름으로 작성하라.
- 파일 이름, `domain`, `topic`에는 소문자 kebab-case를 사용하라.
- 사용자가 목차만 요청하지 않았다면 `index.md`뿐 아니라 각 단계의 완성된 학습 자료도 작성하라.
- 학습 단계 수를 임의로 고정하지 마라. 하나의 중심 질문을 온전히 설명할 수 있는 단위를 기준으로 나누라.
- 비단순 주제는 최소한 개념 오버뷰, 핵심 동작·세부 개념, 응용·판단 층을 서로 다른 문서로 구성하라.
- 난이도 층이나 독립된 소주제가 바뀌면 파일을 분리하고 여러 층을 하나의 거대한 문서로 합치지 마라.
- 대상 독자가 지정되지 않으면 기본 개발 경험과 HTTP·프로세스·코드 실행 지식이 있는 중급 개발자를 기준으로 하라. 필요한 기초는 짧게 복습하되 과정 전체의 수준을 입문에 맞추지 마라.

## 교재 깊이 계약

과정은 용어를 소개하는 데서 끝나지 않고 독자가 다음을 수행하게 해야 한다.

- 메커니즘을 구성 요소와 실행 순서로 재구성한다.
- 변수와 가정을 정의하고 수치 또는 실행 추적으로 결과를 예측한다.
- 공식 계약, 교육용 정신 모델, 현재 구현 세부를 구분한다.
- 동일한 증상을 만드는 여러 원인을 관찰 증거로 구분한다.
- 워크로드와 제약을 보고 설계 대안을 선택한다.
- 실습 결과를 개념과 다시 연결해 설명한다.

각 장은 적어도 `질문 → 필요한 개념 → 유도 또는 실행 추적 → worked example → 경계·반례 → 적용 문제`의 논리적 호를 가져야 한다. 모든 장에 같은 제목을 기계적으로 붙이지 말고 중심 질문에 자연스러운 교재 구조를 사용하라.

다음 형태는 피하라.

- 정의 표와 짧은 코드 뒤 곧바로 요약으로 끝나는 장
- 모든 장이 동일한 8개 섹션과 같은 길이를 갖는 양식 채우기
- 개념 설명보다 학습 목표·내비게이션·체크리스트가 더 긴 문서
- ‘설정에 따라 다르다’고 쓰고 어떤 조건에서 어떻게 달라지는지 설명하지 않는 문장
- 계산 가능한 관계를 수식·단위·수치 예제 없이 직관만으로 설명하는 방식

```text
docs/
├── research/<domain>/<topic>/
│   ├── index.md
│   ├── 01-overview.md
│   ├── 02-<detail>.md
│   └── 03-<application>.md
└── study/<domain>/<topic>/
    ├── index.md
    ├── 01-overview.md
    ├── 02-<detail-lesson>.md
    └── 03-<application-lesson>.md
```

## 구성 워크플로

### 1. 조사 자료를 점검하라

1. 요청한 토픽의 조사 `index.md`를 먼저 읽고 번호가 붙은 모든 조사 문서의 YAML frontmatter, 본문, 참고 자료를 읽어라.
2. 각 문서의 `last_verified`, 대상 버전, 범위와 불확실성을 확인하라.
3. 기존 `docs/study/<domain>/<topic>` 자료가 있으면 먼저 읽고 관련 내용을 보존하라. 새 근거로 수정해야 하는 부분만 명확하게 갱신하라.
4. 기존 `practice/<domain>/<topic>`이 있으면 README와 핵심 테스트·시나리오를 읽고 학습 내용과 실제 관찰값이 일치하는지 확인하라.
5. 다음 항목으로 근거 목록을 내부적으로 구성하라.
   - 제품·프로젝트라면 철학과 연혁, 제품군·에디션 경계, 라이선스 적용 범위, 소유·관리·거버넌스, 검증된 특장점
   - 핵심 개념과 정확한 정의
   - 개념 간 포함, 의존, 인과 관계
   - 입력부터 출력까지의 데이터 또는 제어 흐름
   - 버전별 차이와 적용 조건
   - 예제, 반례, 실패 사례, 트레이드오프
   - 흔한 오해와 확인되지 않은 부분
   - 각 주장에 연결된 직접 출처
6. `조사 문서 → 근거 주장 → 생성할 학습 문서` 대응표를 만들고 각 학습 문서의 `source_documents`에 원본 경로를 남겨라.
7. 핵심 개념을 설명하는 데 필요한 근거가 빠졌으면 본문 작성을 시작하기 전에 조사 자료를 보강하라.

### 2. 선수 지식 그래프를 만들라

1. 학습에 필요한 개념을 한 번에 설명 가능한 최소 단위로 분해하라.
2. `A를 알아야 B를 이해할 수 있다`는 관계를 방향 간선으로 표시하라.
3. 토픽 외부의 선수 지식은 짧게 복습할지 별도 선행 자료로 연결할지 결정하라.
4. 순환 의존이 생기면 공통된 최소 정신 모델을 먼저 소개하고, 뒤 단계에서 정확한 구현을 다시 설명하라.
5. 그래프를 위상 순서로 정렬해 다음 학습 층위로 묶어라.
   - **기초:** 문제 상황, 용어, 최소 정신 모델
   - **구조:** 구성 요소, 책임, 관계
   - **동작:** 단계별 실행, 데이터 흐름, 상태 변화
   - **적용:** 코드, 설정, 실제 시나리오
   - **판단:** 트레이드오프, 실패 분석, 선택 기준
6. `index.md`에 Mermaid 학습 지도를 포함해 선행 관계와 전체 경로를 보여라.

### 3. 학습 순서를 설계하라

조사 문서의 목차를 그대로 사용하지 말고 다음 원칙으로 재배열하라.

- 학습자가 이미 아는 구체적인 문제에서 새로운 추상 개념으로 이동하라.
- 용어를 사용하기 전에 정의하고, 한 단계에서 새로운 추상화 층을 지나치게 많이 도입하지 마라.
- `무엇인가 → 왜 필요한가 → 어떻게 동작하는가 → 언제 선택하는가`를 연결하라.
- 각 단계에 하나의 중심 질문과 관찰 가능한 학습 결과를 부여하라.
- 앞 단계의 핵심 개념을 뒤 단계의 예제와 질문에서 다시 사용해 나선형으로 복습하라.
- 정상 사례 다음에 경계 조건, 반례, 실패 사례를 배치해 개념의 적용 범위를 드러내라.
- API 사용법보다 원리와 데이터 흐름을 먼저 설명하되, 추상 개념은 실행 가능한 예제로 구체화하라.
- 제품·프로젝트의 첫 장은 기능 설명 전에 왜 이 제품이 이런 설계를 택했는지, OSS·cloud·enterprise 중 무엇을 배우는지, 누가 어떤 라이선스로 관리하는지를 독자의 선택 맥락으로 제시하라.

`index.md`의 목차에는 각 단계의 중심 질문, 난이도, 소주제, 선수 지식, 학습 결과, 근거 조사 문서와 시각화 필요 여부를 표시하라. 모든 항목을 실제 학습 문서에 상대 링크로 연결하라. `index.md`는 탐색과 전체 학습 지도만 담당하고 단계별 상세 설명을 중복하지 않게 하라.

### 4. 상세 학습 자료를 작성하라

새 과정을 만들 때 다음 템플릿을 구조적 기준으로 사용하라.

- 과정 목차: `assets/course-index-template.md`
- 단계별 자료: `assets/lesson-template.md`

모든 자리표시자를 실제 내용으로 교체하고 해당하지 않는 선택 섹션은 제거하라. 템플릿을 채우는 데 그치지 말고 학습자가 별도 설명 없이 따라갈 수 있는 완성된 본문을 작성하라.

다음 중 하나라도 해당하면 학습 문서를 분리하라.

- 중심 질문이나 관찰 가능한 학습 결과가 둘 이상이다.
- overview, foundation, detail, application, judgment 사이에서 난이도 층이 바뀐다.
- 소주제의 선수 지식 또는 근거 조사 문서가 달라진다.
- 독립된 예제, 실습 또는 인터랙티브 시각화가 필요하다.
- 다음 문서를 읽지 않고는 현재 문서의 핵심을 설명할 수 없다.

조사 문서 하나를 반드시 학습 문서 하나로 복제하지 마라. 학습 순서에 따라 하나의 조사 문서를 여러 학습 문서로 나누거나 여러 조사 문서를 하나의 밀접한 학습 단계에서 연결할 수 있다. 단, `source_documents`로 대응 관계를 보존하라.

각 단계에 다음 요소를 포함하라.

1. 이전 단계와 현재 단계의 연결
2. 학습자가 먼저 예측해 볼 문제 상황
3. 정확한 기초 개념과 용어 정의
4. 한계가 명시된 최소 정신 모델
5. 내부 동작을 원인과 결과로 연결한 상세 설명
6. 구체적인 값을 사용한 단계별 예제 또는 실행 추적
7. 데이터 흐름, 상태 전이 또는 구조를 보여주는 시각 자료
8. 실제 적용 조건, 트레이드오프, 반례와 흔한 오해
9. 기억 회상, 결과 예측, 새로운 상황에 적용하는 확인 문제
10. 핵심 요약과 다음 단계로 이어지는 질문

제품·프로젝트의 첫 overview 장에는 추가로 다음을 포함하라.

- 탄생 배경과 공식 철학이 구체적 아키텍처·개발 경험에 미친 영향
- OSS core, extension, managed cloud, enterprise edition 등 제품군의 포함 관계와 운영 경계
- 정확한 라이선스 적용 범위와 서비스 약관의 분리
- 최초 개발·현재 소유·실질 유지보수·의사결정 주체의 구분
- 특장점을 대안과 비교하되 유리한 조건, 비용과 한계를 함께 설명하는 선택 기준

심화 주제에는 추가로 다음을 포함하라.

11. 변수·단위·가정을 밝힌 정량 모델 또는 실행 불변식
12. 중간 계산이나 상태를 생략하지 않은 worked example
13. 관찰 증상 → 원인 가설 → 구분 증거 → 다음 조치의 진단 흐름
14. 공식 계약과 현재 구현·버전 세부의 경계
15. 설명, 계산·예측, 디버깅, 설계 판단을 묻는 장말 문제와 풀이 방향

설명을 쉽게 만들기 위해 정확성을 희생하지 마라. 단순화가 필요한 경우 먼저 정신 모델을 제시하고 실제 구현에서 달라지는 지점을 같은 단계 안에서 밝혀라.

## 교재 문체

- 핵심 설명은 완결된 문단으로 전개하고 목록은 분류와 비교에만 사용하라.
- 결론을 먼저 선언한 뒤 왜 그런지 메커니즘과 예제로 증명하라.
- 수식에는 모든 기호와 단위를 정의하고, 식이 성립하는 가정과 깨지는 조건을 바로 설명하라.
- 코드 블록 전에는 무엇을 검증하는지, 뒤에는 실행 중 어떤 상태가 변했는지 설명하라.
- 장 사이에는 앞 장의 결론이 다음 장의 질문으로 이어지는 연결 문단을 둬라.
- 출처 링크는 본문을 대신하지 않는다. 독자가 외부 링크 없이도 핵심 논리를 따라가게 하라.
- 장말 문제는 본문 문장을 그대로 찾는 문제가 아니라 계산, 실행 추적, 진단, 설계 선택을 요구하게 하라.

### 5. 시각 학습을 설계하라

관계를 가장 잘 설명하는 최소한의 표현을 선택하라.

| 학습 관계 | 권장 표현 |
| --- | --- |
| 정확한 값이나 역할의 대응 | 표 |
| 순서가 있는 데이터 이동 | flowchart 또는 sequence diagram |
| 이벤트에 따른 상태 변화 | state diagram |
| 포함, 계층, 선수 관계 | tree 또는 dependency graph |
| 조건별 결과 비교 | 결정 표 또는 분기 흐름 |

세 단계 이상의 흐름이나 보이지 않는 내부 상태가 핵심이면 Mermaid 다이어그램을 포함하라. 학습자가 조작할 때 이해가 깊어지는 내용에는 다음을 갖춘 인터랙션 명세를 작성하라.

- 표현할 상태와 핵심 데이터
- 학습자가 바꿀 입력, 조건, 실행 단계 또는 속도
- 조작 이후의 상태 전이와 애니메이션 순서
- 원인과 결과를 연결하는 시각적 피드백
- 재생, 일시 정지, 한 단계 이동, 초기화
- 키보드 탐색, 모션 감소, 색상 외 구분 방식

애니메이션은 시간, 순서, 동시성, 인과관계를 설명할 때만 사용하라.

### 6. 근거를 연결하라

- 조사 문서의 공식 출처를 해당 사실과 가까운 위치에 직접 연결하라.
- `source_documents` frontmatter에 사용한 조사 문서의 프로젝트 상대 경로를 기록하라.
- 학습을 위해 추가한 설명이 원문 근거보다 강한 주장으로 바뀌지 않았는지 확인하라.
- 여러 조사 문서의 관점이 다르면 하나로 뭉개지 말고 적용 조건과 차이를 설명하라.
- 학습 단계 끝의 `참고 자료`에는 그 단계에서 실제 사용한 자료만 포함하라.

## 완료 검증

완료 전에 다음 항목을 모두 확인하라.

- `index.md`의 목차, 파일명, 링크와 각 문서의 `order`가 일치하는가?
- `docs/study/<domain>/<topic>`이 입력 `docs/research/<domain>/<topic>`과 정확히 같은 주제 경로를 사용하는가?
- `index.md`가 상세 내용을 종합하지 않고 학습 지도와 문서 탐색만 담당하는가?
- 순서 번호가 `01`부터 중복이나 누락 없이 이어지는가?
- 개념 오버뷰, 세부 개념·동작, 응용·판단이 난이도와 소주제에 따라 별도 문서로 나뉘었는가?
- 모든 학습 문서가 관련 조사 문서를 `source_documents`로 추적할 수 있는가?
- 어떤 단계도 뒤 단계의 설명을 선수 지식으로 요구하지 않는가?
- 각 단계가 하나의 중심 질문에 답하면서 기초 개념부터 상세 동작까지 연결하는가?
- 조사 자료에 없는 사실을 출처 없이 새로 단정하지 않았는가?
- 버전과 조건, 추론, 정신 모델, 비유의 표시가 보존됐는가?
- 예제가 추상 설명을 실제 값과 실행 과정으로 보여주는가?
- 회상, 예측, 적용 유형의 확인 문제가 모두 있는가?
- Mermaid, 코드 블록, 내부 링크와 상대 경로가 유효한가?
- 시각화와 애니메이션이 학습 목표에 직접 기여하는가?
- 템플릿 자리표시자와 빈 필수 섹션이 남아 있지 않은가?
- 개념의 정확한 정의 이후 메커니즘과 인과관계가 충분히 전개됐는가?
- 적용 가능한 장에 변수·단위·가정·유도와 worked example이 있는가?
- 증상에서 원인을 구분하는 진단 절차와 반례가 있는가?
- 장말 문제가 설명, 예측·계산, 진단, 설계 판단을 모두 요구하는가?
- 고정 템플릿을 반복한 요약집이 아니라 장마다 자연스러운 논증 흐름을 가진 교재인가?
- 제품·프로젝트라면 첫 overview 장이 철학, 제품군, 라이선스, 소유·관리·거버넌스와 조건부 특장점을 학습자의 선택 맥락으로 전달하는가?

최종 응답에는 읽은 조사 문서, 생성하거나 갱신한 과정과 학습 자료 경로, 전체 단계 수, 대상 수준, 발견한 근거 부족이나 남은 불확실성을 간결하게 보고하라.
