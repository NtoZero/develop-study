# Tech Research

공신력 있는 최신 근거를 기술 학습에 적합한 개념 모델, 데이터 흐름, 예제와 시각화 설계로 변환하라. 결과를 프로젝트의 `docs/research` 아래에 재사용 가능한 Markdown 문서로 남겨라.

## 기본값

- 사용자가 별도로 지정하지 않으면 한국어로 작성하라.
- 주제 경로를 `docs/research/<domain>/<topic>/`으로 구성하라.
- `domain`과 `topic`에는 소문자 kebab-case만 사용하라.
- 같은 주제의 후속 학습 자료가 `docs/study/<domain>/<topic>/`에 정확히 대응할 수 있도록 `domain`과 `topic` 경로를 안정적으로 유지하라.
- `index.md`에는 조사 범위, 문서 지도, 선수 관계와 검증 상태만 작성하고 상세 내용을 종합해 넣지 마라.
- 실제 조사 내용은 `01-overview.md`, `02-<subtopic>.md`, `03-<application>.md`처럼 순서와 범위가 드러나는 여러 문서로 분리하라.
- 비단순 주제는 최소한 개념 오버뷰, 핵심 동작·세부 개념, 응용·판단 층을 서로 다른 문서로 나누라. 주제의 크기에 따라 세부 문서를 더 나누되 고정된 문서 수를 맞추기 위해 내용을 억지로 합치거나 쪼개지 마라.
- 하나의 내용 문서는 하나의 중심 질문, 하나의 난이도 층, 밀접한 하나의 소주제 묶음만 다루게 하라.
- `domain`은 우선 `cs`, `fe`, `be`, `db`, `infra` 중에서 선택하라. 경계에 걸친 주제는 핵심 학습 관점에 따라 하나를 주 영역으로 정하고 나머지는 `tags`에 기록하라.
- 현재 날짜를 `last_verified`로 기록하라.
- 사용자가 버전을 지정하지 않으면 프로젝트 설정에서 버전을 확인하라. 확인할 수 없고 버전 차이가 결론을 바꿀 때만 사용자에게 질문하라. 그 외에는 조사한 버전과 가정을 문서에 명시하고 진행하라.

대표적인 분류 예시는 다음과 같다.

| 주제 | domain 예시 |
| --- | --- |
| 운영체제, 네트워크 원리, 자료구조, 알고리즘 | `cs` |
| 브라우저, React, CSS, 웹 접근성 | `fe` |
| Spring, API, 인증, 메시징, 분산 시스템 | `be` |
| PostgreSQL, 인덱스, 트랜잭션, 쿼리 최적화 | `db` |
| Linux 운영, 컨테이너, Kubernetes, k6, 관측 가능성 | `infra` |

```text
docs/research/<domain>/<topic>/
├── index.md
├── 01-overview.md
├── 02-<core-subtopic>.md
├── 03-<mechanism-or-detail>.md
└── 04-<application-or-advanced>.md
```

## 조사 워크플로

### 1. 범위를 고정하라

1. 사용자의 핵심 질문, 대상 독자, 선수 지식, 기대 깊이를 한 문장씩 정리하라.
2. 조사 대상의 버전, 런타임, 운영 환경처럼 결론에 영향을 주는 조건을 식별하라.
3. 기존 `docs/research/<domain>/<topic>` 문서와 프로젝트 의존성을 먼저 확인하라. 기존 문서가 있으면 중복 생성하지 말고 사용자의 의도에 맞게 갱신하라.
4. 조사 중 답해야 할 하위 질문을 개념 오버뷰, 세부 동작, 응용·판단 층으로 분류하라.
5. 각 하위 질문을 하나의 문서 또는 밀접한 문서 묶음에 배정하고 조사 순서, 난이도, 선수 관계를 먼저 정하라.
6. 하나의 문서에 서로 다른 난이도 층이나 독립된 중심 질문이 들어가면 작성 전에 파일을 분리하라.

### 2. 출처를 수집하라

기술 주장에는 가능한 한 1차 자료를 사용하고 다음 우선순위를 적용하라.

1. 표준 문서, 공식 명세, RFC, 공식 언어·플랫폼 문서
2. 공식 프로젝트·벤더 문서, 공식 저장소와 릴리스 노트
3. 원 논문, 학회 또는 연구기관 자료
4. 공식 엔지니어링 블로그나 설계 문서

라이브러리, 프레임워크, SDK, API, CLI를 조사할 때는 다음 순서를 지켜라.

1. Context7의 `resolve-library-id`로 공식 프로젝트를 식별하라.
2. 사용자의 버전과 일치하는 라이브러리 ID가 있으면 해당 버전을 선택하라.
3. Context7의 `query-docs`로 질문을 한 가지 개념씩 구체적으로 조회하라.
4. Context7 응답에 포함된 원본 공식 문서를 직접 인용 출처로 사용하라. Context7 자체를 기술 주장의 출처로 표기하지 마라.
5. 웹 검색으로 공식 명세, 릴리스 노트, 관련 계층의 문서를 보완하고 최신성을 교차 검증하라.

Context7가 노출되지 않거나 조회에 실패하면 공식 웹 문서로 조사를 계속하고 해당 제한을 최종 응답에 알려라. 충분한 1차 자료를 찾지 못하면 추측으로 빈틈을 채우지 말고 확인된 범위와 남은 불확실성을 문서에 기록하라.

검색 결과 요약문, 출처가 불명확한 블로그, 복제 문서, SEO 콘텐츠를 근거로 사용하지 마라. 좁은 사실은 하나의 권위 있는 표준 문서로 충분할 수 있다. 여러 시스템에 걸친 동작, 비교, 논쟁의 여지가 있는 주장은 서로 독립적인 1차 출처 둘 이상으로 검증하라.

### 3. 근거를 검증하라

- 각 핵심 주장에 대해 `주장 → 출처 → 적용 버전 또는 날짜`를 내부적으로 대응시켜라.
- 문서의 게시일보다 해당 기능이 적용되는 버전과 최신 개정 여부를 우선 확인하라.
- 서로 충돌하는 자료가 있으면 최신 공식 명세와 대상 버전에 가까운 자료를 우선하되, 충돌과 판단 근거를 문서에 남겨라.
- 확인되지 않은 내용을 사실처럼 쓰지 마라. 추론은 `추론`, 학습을 위한 단순화는 `정신 모델`, 비유는 `비유`로 명시하라.
- 시간이나 버전에 따라 변할 수 있는 주장에는 가까운 위치에 직접 링크를 배치하라.
- 검색 결과 페이지가 아닌 실제 문서 페이지에 링크하라.
- 긴 원문 인용을 피하고 의미를 정확히 바꾸어 설명하라. 코드나 짧은 문구를 인용할 때도 출처를 표시하라.

### 4. 학습 모델을 설계하라

사실을 나열하지 말고 다음 순서로 이해를 구축하라.

1. 이 기술이 해결하는 문제와 등장 배경
2. 가장 작은 유효한 정신 모델
3. 구성 요소와 각 요소의 책임
4. 입력에서 출력까지의 단계별 데이터 또는 제어 흐름
5. 조건을 바꿨을 때 달라지는 결과와 그 원인
6. 트레이드오프, 경계 조건, 실패 사례와 흔한 오해
7. 실제 적용 예제와 스스로 답할 수 있는 확인 질문

세 단계 이상의 흐름, 상태 전이, 계층 관계가 이해에 중요하면 Mermaid 다이어그램을 포함하라. 시각화가 도움이 되는 주제에는 구현 가능한 인터랙션 명세를 작성하라.

- **상태:** 화면이 표현해야 할 핵심 상태와 데이터
- **조작:** 학습자가 변경할 입력, 실행 순서, 속도 또는 조건
- **전이:** 조작 후 발생하는 상태 변화와 애니메이션 순서
- **피드백:** 학습자가 원인과 결과를 연결하도록 보여줄 값과 설명
- **접근성:** 키보드 조작, 모션 감소, 색상 외 구분 방법

애니메이션을 장식 목적으로 제안하지 마라. 인과관계, 동시성, 순서 또는 보이지 않는 데이터 이동을 설명할 때만 사용하라.

### 5. 문서를 작성하라

새 주제를 만들 때 다음 템플릿을 구조적 기준으로 사용하라.

- 조사 목차: `assets/research-index-template.md`
- 개별 조사 문서: `assets/research-template.md`

먼저 `index.md`를 만들고, 모든 실제 조사 문서를 번호가 붙은 별도 파일로 작성하라. 모든 자리표시자를 실제 값으로 교체하고 주제에 해당하지 않는 선택 섹션은 제거하라.

다음 중 하나라도 해당하면 문서를 분리하라.

- 중심 질문이 둘 이상이다.
- 기초 설명에서 내부 구현 또는 응용으로 난이도가 바뀐다.
- 소주제의 선수 지식이나 적용 버전이 다르다.
- 데이터 흐름이나 인터랙티브 시각화 모델이 독립적으로 성립한다.
- 한 부분만 갱신하거나 재검증할 가능성이 높다.

다음 작성 규칙을 적용하라.

- 모든 내용 문서의 frontmatter에 `order`, `level`, `subtopic`을 기록하라.
- `index.md`의 문서 순서, 파일명, 중심 질문, 난이도와 실제 파일을 일치시켜라.
- 제목과 첫 문단만 읽어도 문서의 범위와 핵심 결론을 알 수 있게 하라.
- 전문 용어를 처음 사용할 때 정의하라.
- 코드 예제는 최소 실행 단위로 작성하고 대상 버전을 표시하라.
- 사실과 가까운 위치에 `[설명적인 출처명](직접 URL)` 형식으로 인용하라.
- 문서 끝의 `참고 자료`에 실제 사용한 자료만 모으고 소유 기관, 적용 버전, 확인일을 기록하라.
- 출처만 추가하고 본문에서 사용하지 않은 자료를 참고 목록에 넣지 마라.
- 기존 파일을 갱신할 때 사용자의 관련 내용을 보존하고, 근거 없이 전면 재작성하지 마라.

## 완료 검증

완료 전에 다음 항목을 모두 확인하라.

- 주제 디렉터리가 정확히 `docs/research/<domain>/<topic>/`에 있는가?
- 같은 주제를 위한 `docs/study` 경로가 동일한 `domain/topic`으로 대응 가능한가?
- `index.md`가 상세 내용을 중복하지 않고 모든 조사 문서를 올바른 순서와 링크로 안내하는가?
- 내용이 단일 종합 문서에 몰리지 않고 개념 오버뷰, 세부 소주제, 응용·판단 층으로 분리됐는가?
- 번호가 `01`부터 중복이나 누락 없이 이어지고 파일명에 소주제가 드러나는가?
- YAML frontmatter의 `domain`, `topic`, `order`, `level`, `subtopic`, `last_verified`가 실제 조사와 일치하는가?
- 템플릿 자리표시자와 빈 필수 섹션이 남아 있지 않은가?
- 핵심 주장마다 직접 연결된 공신력 있는 출처가 있는가?
- 버전 의존적인 내용을 일반적인 사실로 과장하지 않았는가?
- 사실, 추론, 정신 모델, 비유가 구분되어 있는가?
- Mermaid와 코드 블록의 문법 및 용어가 본문과 일치하는가?
- 시각화 제안이 학습 목표와 인과관계를 실제로 설명하는가?
- 링크 제목만으로 출처의 성격을 이해할 수 있는가?

최종 응답에는 생성하거나 갱신한 주제 디렉터리와 문서 목록, 난이도 층, 조사 기준 버전과 확인일, 사용한 핵심 출처의 수, 남은 불확실성을 간결하게 보고하라.
