# Visualize Content

검증된 학습 원문을 사용자가 직관적으로 이해하고 순차적으로 탐색하며 직접 조작할 수 있는 Next.js 학습 경험으로 구현하라. 주제마다 고유한 인과관계와 상태를 인터랙션의 중심에 두고, 일반적인 카드·탭·슬라이드 조합만으로 완료하지 마라.

## 필수 결과

완성된 경험이 다음 질문에 모두 `예`라고 답하게 하라.

- 첫 화면에서 무엇을 배우고 현재 어디에 있는지 이해할 수 있는가?
- 기초부터 다음 단계까지 순서대로 이동하거나 학습 지도를 통해 원하는 단계로 갈 수 있는가?
- 학습자가 주제에 의미 있는 입력이나 조건을 직접 바꿀 수 있는가?
- 입력 변화가 내부 상태와 결과를 어떻게 바꾸는지 즉시 관찰할 수 있는가?
- 시각화가 원문의 정확한 개념, 버전, 경계 조건을 보존하는가?
- 키보드, 터치, 작은 화면, 모션 감소 환경에서도 같은 내용을 학습할 수 있는가?

## 역할과 경계

- `docs/study/<domain>/<topic>/index.md`와 번호가 붙은 단계별 문서를 학습 구조와 표현의 기준으로 사용하라.
- 정밀한 동작 근거가 필요하면 연결된 `docs/research/<domain>/<topic>` 문서까지 읽어라.
- 대응하는 `practice/<domain>/<topic>`이 있으면 실제 실행 명령, 입력, 관찰값과 성공·실패 기준을 읽고 시각화의 실습 안내와 일치시켜라.
- 학습 원문이 없거나 미완성이면 기억으로 콘텐츠를 만들지 마라. `$make-study-content`로 먼저 구성하고, 근거 자체가 부족하면 `$tech-research`로 보강하라.
- 구현 대상은 저장소 루트의 `visualization/` 안에 이미 존재하는 Next.js 앱이다. 해당 디렉터리나 `package.json`이 없으면 임의로 새 앱을 만들지 말고 무엇이 없는지 보고하라. 사용자가 명시적으로 앱 생성을 요청했을 때만 스캐폴딩하라.
- 기존 디자인 시스템, 라우팅, 콘텐츠 로더, 상태 관리, 테스트 방식과 사용자 변경을 보존하라.
- 로컬 기본 포트로 `3113`을 사용하되 실제 `package.json` 스크립트와 실행 중인 서버를 먼저 확인하라.

## 구현 워크플로

### 1. 앱과 원문을 탐색하라

작업 전에 다음을 확인하라.

1. 루트와 `visualization/` 아래의 `AGENTS.md` 및 프로젝트 지침
2. `package.json`, lockfile, Next.js·React·TypeScript 버전과 실행 스크립트
3. `app` 또는 `src/app`, Pages Router 여부, 라우트와 레이아웃 규칙
4. Tailwind 설정, 전역 스타일, 디자인 토큰, 폰트, 공용 컴포넌트
5. 설치된 아이콘, 차트, 애니메이션, 상태 관리, 테스트 라이브러리
6. 관련 `docs/study` 과정의 목차와 모든 단계별 원문
7. 연결된 연구 문서, 대상 버전, 출처와 남은 불확실성
8. 대응하는 `practice/<domain>/<topic>`의 README, 실행 명령과 검증 결과
9. 같은 영역에 이미 구현된 학습 페이지의 URL·파일·상호작용 패턴

lockfile로 패키지 관리자를 판단하라. 기존 도구로 충분하면 새 의존성을 추가하지 마라. 새 의존성이 실제로 필요하면 선택 이유와 번들·유지보수 비용을 확인하고 프로젝트의 패키지 관리자를 사용하라.

Next.js, React, Tailwind 또는 설치된 시각화·애니메이션 라이브러리의 API를 사용할 때는 Context7에서 공식 프로젝트와 설치 버전을 먼저 식별하고 필요한 개념을 구체적으로 조회하라. Context7가 없으면 해당 프로젝트의 공식 문서를 직접 확인하고, 기억만으로 버전 의존 API를 작성하지 마라.

### 2. 학습 내용을 인터랙션 모델로 바꾸라

다음 대응표를 내부적으로 만든 뒤 코드를 작성하라.

```text
학습 단계 → 중심 질문 → 표현할 상태 → 사용자 조작 → 상태 전이 → 관찰 피드백 → 근거 문서
```

각 단계에서 다음을 추출하라.

- 학습자가 설명하거나 예측해야 하는 핵심 결과
- 구성 요소, 데이터, 상태, 이벤트와 관계
- 시간 순서, 분기, 동시성, 계층 또는 트레이드오프
- 조작 가능한 실제 변수와 허용 범위
- 정상 결과, 경계 조건, 실패 상태와 흔한 오해
- 교육용 단순화와 실제 구현이 달라지는 지점

인터랙션 유형을 고르기 전에 `references/interaction-patterns.md`를 전부 읽고 주제에 가장 가까운 패턴을 선택하라. 한 가지 주 인터랙션을 중심에 두고 꼭 필요한 보조 표현만 추가하라.

다음 계약을 명시적으로 정의하라.

- **상태:** 화면과 도메인 모델이 보유하는 값
- **행동:** 학습자가 실행할 수 있는 의미 있는 조작
- **전이:** 행동 전후에 변하는 상태와 불변 조건
- **피드백:** 변화의 원인과 결과를 연결해 보여주는 텍스트·형태·수치
- **복구:** 초기화, 다시 실행, 이전 단계와 유효하지 않은 입력 처리

도메인 계산과 애니메이션 상태를 분리하라. 같은 입력은 같은 결과를 만들게 하고, 임의의 타이머 체인에 핵심 동작을 의존시키지 마라. 실제 시스템을 완전히 재현하지 않는다면 `교육용 모델`의 범위와 한계를 화면에 표시하라.

### 3. 순차 탐색 구조를 설계하라

- 과정 개요에 전체 학습 지도, 선수 관계, 현재 위치와 완료 상태를 보여라.
- 각 단계에 이전·다음 이동, 전체 목차 복귀, 현재 단계 표시를 제공하라.
- 긴 단계는 개념 단위의 작은 장면으로 나누고 단계 이동 또는 스크롤 위치를 명확히 표시하라.
- 순차 애니메이션에는 재생, 일시 정지, 한 단계 이전·다음, 초기화와 필요할 때 속도 조절을 제공하라.
- 페이지를 새로 열거나 공유해도 학습 단계에 다시 접근할 수 있도록 기존 라우트 규칙에 맞는 URL, 동적 세그먼트, query 또는 hash를 사용하라.
- 진행률은 시간이나 스크롤 길이가 아니라 실제로 방문하거나 완료한 학습 단계에 기반하라. 저장 방식은 기존 프로젝트의 정책을 따르라.
- 순서를 강제해 탐색을 막지 마라. 권장 순서는 명확히 보여주되 이미 아는 단계로 이동할 수 있게 하라.

### 4. 주제에 근거한 시각 방향을 정하라

코드 전에 짧은 디자인 계획을 세워라.

1. **주제와 사용자:** 구체적인 기술 주제, 학습자 수준, 페이지의 한 가지 핵심 역할
2. **색상:** 기존 토큰을 우선한 4~6개의 역할별 색상과 의미
3. **타이포그래피:** 제목, 본문, 코드·데이터에 사용할 기존 폰트와 위계
4. **레이아웃:** 학습 순서와 상태 관계를 반영한 화면 구조
5. **시그니처:** 해당 주제의 동작을 가장 잘 드러내는 하나의 기억할 만한 인터랙션

주제의 실제 언어와 구조에서 시각적 모티프를 얻어라. 데이터베이스 인덱스, 네트워크 패킷, 이벤트 큐처럼 학습 대상 자체를 화면의 구조로 사용하라. 어디에나 적용 가능한 대시보드 카드 그리드, 장식용 그라데이션, 근거 없는 숫자, 동일한 영웅 섹션을 반복하지 마라. 대담한 선택은 시그니처 인터랙션 한 곳에 집중하고 나머지는 조용하고 정확하게 구성하라.

인터페이스 문구는 구현 용어가 아니라 학습자가 조작하고 관찰하는 대상의 이름으로 작성하라. 버튼 이름과 결과 피드백에 같은 동사를 사용하라.

### 5. Next.js 구조에 맞게 구현하라

실제 프로젝트 구조를 따르고 새 구조를 병렬로 만들지 마라.

App Router를 사용하는 경우 다음을 기본으로 하라.

- 라우트 `page.tsx`와 정적 설명은 Server Component로 유지하라.
- 상태, 이벤트 핸들러, 브라우저 API가 필요한 최소 인터랙티브 영역만 Client Component로 분리하라.
- 서버에서 클라이언트로 넘기는 학습 데이터는 직렬화 가능한 타입으로 제한하라.
- 도메인 계산을 순수 함수와 명시적 타입으로 분리해 UI 없이 테스트할 수 있게 하라.
- 전체 페이지에 불필요하게 `'use client'`를 선언하지 마라.

Pages Router를 사용한다면 기존 방식을 유지하고 App Router 마이그레이션을 작업 범위에 포함하지 마라.

구조를 선택할 때 다음 원칙을 적용하라.

- 기존 Markdown·MDX 콘텐츠 파이프라인이 있으면 재사용하고 원문을 긴 JSX 문자열로 복제하지 마라.
- 콘텐츠 파이프라인이 없으면 타입이 지정된 로컬 데이터 모델과 `sourceDocuments` 메타데이터를 만들고 원문과의 연결을 남겨라.
- 공용 내비게이션, 레이아웃, 컨트롤은 재사용하되 주제별 도메인 모델과 시각화는 기능 단위로 응집하라.
- TypeScript의 안전성을 유지하고 `any`로 상태 모델을 우회하지 마라.
- Tailwind와 기존 토큰을 우선 사용하고 전역 CSS를 불필요하게 확장하지 마라.
- 설치된 애니메이션 도구가 있으면 기존 방식을 따르고, 단순 전환은 CSS와 브라우저 기능으로 해결하라.

### 6. 접근성과 반응형 동작을 보장하라

- 모든 조작을 키보드로 실행하고 현재 포커스를 볼 수 있게 하라.
- 아이콘 버튼에 접근 가능한 이름을 제공하고 활성·비활성·선택 상태를 텍스트나 형태로도 구분하라.
- 색상만으로 노드, 성공, 실패, 흐름을 구분하지 마라.
- 상태 변화의 핵심 결과를 텍스트로 함께 제공하고 필요한 경우 간결한 live region을 사용하라.
- `prefers-reduced-motion`에서는 자동 재생과 큰 이동을 끄고 즉시 전환 또는 단계 이동으로 대체하라.
- 작은 화면에서는 정보 순서와 조작 가능성을 보존하라. 단순히 전체 캔버스를 축소하지 말고 세로 흐름, 드릴다운 또는 스크롤 가능한 축으로 재배치하라.
- 차트와 복잡한 SVG에는 요약, 레이블, 수치 또는 표 형태의 대안을 제공하라.
- 터치 대상과 드래그 조작에 충분한 크기와 비드래그 대체 조작을 제공하라.

### 7. 실제 동작을 검증하라

프로젝트가 제공하는 스크립트와 도구를 사용해 위험에 비례하여 검증하라.

1. 도메인 순수 함수와 상태 전이의 경계값을 테스트하라.
2. 기존 테스트 환경이 있으면 단계 이동, 초기화, 잘못된 입력과 핵심 학습 피드백을 컴포넌트 테스트로 확인하라.
3. lint, TypeScript 검사, 관련 테스트와 production build를 실행하라.
4. 개발 서버를 프로젝트 기본 포트 `3113`에서 실행하거나 이미 실행 중인 서버를 사용하라. 포트를 점유한 알 수 없는 프로세스를 종료하지 마라.
5. 브라우저에서 직접 라우트를 열어 데스크톱과 작은 화면을 확인하라.
6. 모든 주요 조작, 이전·다음 이동, 초기화, 새로고침과 직접 URL 접근을 실행하라.
7. 키보드 탐색, 포커스, 모션 감소 동작, 콘솔 오류와 레이아웃 넘침을 확인하라.
8. 시각적 위계와 애니메이션 순서를 스크린샷으로 다시 비평하고, 주제와 무관한 장식을 제거하라.

실패한 검사를 생략한 채 완료했다고 말하지 마라. 수정할 수 없는 환경 문제는 실행한 검사와 함께 정확히 보고하라.

## 완료 기준

- 원문의 모든 학습 단계가 구현되거나 명시적으로 연결됐는가?
- 학습 지도, 현재 위치, 이전·다음 경로와 직접 접근 URL이 일치하는가?
- 주제에 의미 있는 조작이 실제 도메인 상태와 결과를 바꾸는가?
