# Prepare Practice

조사와 학습 자료에서 검증해야 할 동작을 추출해 로컬에서 안전하게 실행하고 결과를 관찰할 수 있는 실습 환경으로 구현하라.

## 핵심 규칙

- 모든 실습 루트는 `practice/<domain>/<topic>/`을 사용하라.
- `docs/research/<domain>/<topic>/`, `docs/study/<domain>/<topic>/`, `practice/<domain>/<topic>/`의 경로를 정확히 대응시켜라.
- 저장소 루트에 `<topic>/`, `<framework>/` 같은 독립 실습 디렉터리를 새로 만들지 마라.
- `visualization/`은 개념 시각화 앱, `practice/`는 실제 런타임·테스트·도구 실행 환경으로 책임을 분리하라.
- 기존 실습이 다른 위치에 있으면 내용을 보존해 대응 경로로 이동하고 모든 문서·명령·링크를 함께 갱신하라.
- 실습은 기본적으로 로컬 또는 격리된 컨테이너만 대상으로 하라. 사용자의 명시적 허가 없이 운영·외부 서비스에 부하, 변경, 데이터 생성을 수행하지 마라.

## 입력 확인

작업 전에 다음을 순서대로 확인하라.

1. 루트와 대상 경로의 `AGENTS.md` 및 프로젝트 지침
2. `docs/research/<domain>/<topic>/index.md`와 관련 조사 문서 전체
3. `docs/study/<domain>/<topic>/index.md`와 단계별 학습 문서 전체
4. 기존 `practice/<domain>/<topic>/`와 관련 코드·테스트·설정
5. `visualization/`에서 실습 명령이나 결과를 참조하는 부분
6. 사용 가능한 런타임, 패키지 관리자, 컨테이너 도구와 버전

`domain`과 `topic`이 문서 경로에서 명확하면 그대로 사용하라. 새 토픽이면 저장소의 기존 slug 규칙을 따르고, 경로 선택이 실제 학습 분야를 바꿀 정도로 모호할 때만 사용자에게 확인하라.

## 실습 설계

학습 자료에서 다음 항목을 뽑아 실습 행렬을 먼저 세워라.

| 항목 | 질문 |
| --- | --- |
| 학습 목표 | 실행 후 무엇을 설명하거나 예측할 수 있어야 하는가? |
| 가설 | 어떤 입력에서 어떤 결과가 나와야 하는가? |
| 조작 변수 | 학습자가 한 번에 무엇을 바꾸는가? |
| 관찰값 | 로그, 응답, 메트릭, 상태, 테스트 결과 중 무엇을 보는가? |
| 성공·실패 기준 | 정상 성공과 학습용 의도적 실패를 어떻게 구분하는가? |
| 안전 경계 | 데이터, 네트워크, 리소스, 종료 조건을 어떻게 제한하는가? |

기초 확인에서 시작해 한 번에 한 변수만 바꾸고, 마지막에 응용 과제로 확장하라. 단순한 완성 코드 복사보다 예측 → 실행 → 관찰 → 설명의 순환을 우선하라.

## 구현 절차

1. 대상 `practice/<domain>/<topic>/` 경로를 확정하라.
2. 해당 생태계의 표준 프로젝트 구조와 테스트 도구를 선택하라.
3. 공식 문서 또는 조사 문서에 확인된 버전을 사용하고 manifest, wrapper, image tag 등 재현 가능한 위치에 기록하라.
4. 외부 의존성을 최소화한 로컬 대상, fixture, fake 또는 컨테이너를 준비하라.
5. smoke 또는 최소 정상 경로를 먼저 구현하라.
6. 개념 차이를 드러내는 세부 실습과 의도적 실패 실습을 추가하라.
7. 자동 검증 가능한 테스트·assertion·threshold를 두고 예상 종료 상태를 명시하라.
8. `assets/practice-readme-template.md`를 바탕으로 실행·관찰·복구 방법을 문서화하라.
9. `docs/study`와 `visualization`의 실습 링크와 명령을 새 경로에 맞게 갱신하라.
10. 깨끗한 환경을 가정해 설치, 최소 실행, 테스트, 의도적 실패를 실제로 검증하라.

## 기술별 구조 선택

- Node.js·브라우저 도구: 대상 토픽 안에 `package.json`, 소스, 테스트, 필요 시 lockfile을 둬라.
- Java·Spring: Gradle 또는 Maven 표준 구조와 wrapper를 우선하고, 학습 테스트를 `src/test`에 소주제별로 분리하라.
- Python·AI 라이브러리: `pyproject.toml`과 테스트 디렉터리를 우선하고 환경변수 예시는 `.env.example`만 제공하라.
- Infra·DB·부하 도구: Docker Compose, 로컬 fixture, seed와 정리 명령을 사용해 외부 시스템 의존을 줄여라.
- 여러 런타임이 필요한 경우에도 모든 파일을 하나의 `practice/<domain>/<topic>/` 아래에 두고 `target/`, `scenarios/`, `fixtures/`처럼 책임별로 나눠라.

표준 구조가 학습 목표와 충돌하지 않는 한 독자적인 빌드 체계나 범용 모노레포를 새로 만들지 마라.

## README 요구사항

README에는 최소한 다음을 포함하라.

- 어떤 조사·학습 문서를 검증하는지
- 필요한 도구와 확인된 버전
- 최초 설치와 실행 명령
- 실습 순서별 가설, 조작 변수, 관찰값, 예상 결과
- 정상 성공과 의도적 실패의 종료 상태
- 직접 바꿔 볼 과제
- 중단·정리·초기화 방법
- 외부·운영 시스템 사용 금지 등 안전 경계
- 공식 출처와 대응 학습 자료 링크

경로를 포함한 명령은 저장소 루트에서 실행하는지 토픽 디렉터리에서 실행하는지 명시하라.

## 안전과 비밀정보

- 비밀값을 소스, 문서, fixture, Git 추적 파일에 넣지 마라.
- 환경변수가 필요하면 `.env.example`에 키 이름과 안전한 가짜 값만 제공하라.
- 데이터 변경 실습은 격리된 데이터와 멱등적인 정리 절차를 제공하라.
- 부하·장애·보안 실습은 기본 대상이 localhost인지 실행 전에 검사하라.
- 비용이 발생하거나 외부 상태를 바꾸는 실행은 사용자의 명시적 승인을 받아라.

## 검증

가능한 범위에서 다음을 모두 수행하라.

1. manifest와 설정 구문 검사
2. formatter 또는 정적 검사
3. 단위·학습 테스트
4. 최소 정상 실습 실행
5. 의도적 실패가 문서화한 방식으로 실패하는지 확인
6. 종료 후 프로세스·컨테이너·임시 데이터 정리
7. Markdown 상대 링크와 경로 명령 확인

도구가 없거나 권한이 필요한 검증은 조용히 생략하지 말고 정확한 미검증 범위와 사용자가 실행할 명령을 남겨라.

## 완료 기준

- 실습이 `practice/<domain>/<topic>/`에만 존재한다.
- research, study, practice가 같은 `domain/topic`으로 대응한다.
- 새 환경에서 README 순서대로 실행할 수 있다.
- 학습자가 입력을 바꾸고 결과를 관찰할 수 있다.
- 예상 성공과 예상 실패를 자동으로 구분한다.
- 외부 시스템, 비밀정보, 데이터 정리에 대한 안전 경계가 명확하다.
- 관련 문서와 시각화의 링크·명령이 현재 경로와 일치한다.
